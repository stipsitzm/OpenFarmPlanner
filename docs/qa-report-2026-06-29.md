## 4. Mittlere Bugs

### BUG-M01 – Ungültige Routen rendern leere Seite ohne Fehlermeldung

**Priorität:** Medium  
**Bereich:** Routing / Fehlerbehandlung  

**Reproduktionsschritte:**
1. Eingeloggt sein
2. `localhost:5173/app/nonexistent-page-12345` direkt in der Adressleiste eingeben

**Erwartetes Verhalten:**  
Anzeige einer benutzerfreundlichen "Seite nicht gefunden"-Meldung oder automatische Weiterleitung zur Startseite.

**Tatsächliches Verhalten:**  
Die App bleibt auf der ungültigen URL stehen und rendert eine nahezu leere Seite (nur ~108 Zeichen DOM-Text, keine Fehlermeldung, keine Orientierung für den Nutzer). Kein 404-Hinweis, kein Redirect.

**Vermutete Ursache:**  
React Router hat keine Catch-All-Route (`*`) für unbekannte Pfade unterhalb von `/app/`. Die Shell-Komponente rendert ohne passende Route einfach leer.

### BUG-M02 – RegisterPage: Hardcoded englische aria-labels auf Passwort-Toggle-Buttons (Regression)

**Priorität:** Medium (Barrierefreiheit)  
**Bereich:** Registrierung  
**Status:** Regression – war in `qa-report-2026-06-20-new-user.md` als Problem dokumentiert

**Reproduktionsschritte:**
1. `/register` aufrufen
2. Screenreader aktivieren oder aria-label der Augen-Buttons prüfen (DevTools → Elements → `aria-label`)

**Erwartetes Verhalten:**  
Die Buttons tragen deutsche Labels aus der i18n-Übersetzung (`t('auth:register.showPassword')` / `t('auth:register.hidePassword')`).

**Tatsächliches Verhalten:**  
Beide Passwort-Toggle-Buttons (`Passwort` und `Passwort bestätigen`) haben hardcoded englische aria-labels:
```
aria-label="Show password"  (oder "Hide password")
```

**Code-Fundstelle:**  
`frontend/src/pages/auth/RegisterPage.tsx:146` und `:171`

**Vergleich:** `LoginPage.tsx` verwendet die gleiche Funktion korrekt:
```tsx
// LoginPage.tsx (korrekt)
aria-label={t(showPassword ? 'auth:login.hidePassword' : 'auth:login.showPassword')}

// RegisterPage.tsx (fehlerhaft)
aria-label={showPassword ? 'Hide password' : 'Show password'}
```

Die Übersetzungsschlüssel existieren in `de/auth.json` unter `register.showPassword` / `register.hidePassword` — sie werden nur nicht verwendet.

**Vermutete Ursache:**  
Die LoginPage wurde als Vorlage genutzt, aber bei der RegisterPage wurden die `t()`-Aufrufe nicht übernommen. Der Fix aus dem vorherigen Zyklus wurde entweder nie eingecheckt oder danach wieder verloren.

## 5. Niedrige Bugs

### BUG-L01 – CommandProvider-Tipp-Banner: hardcoded deutscher Text ohne i18n

**Priorität:** Low  
**Bereich:** Keyboard-Shortcuts / CommandPalette  

**Reproduktionsschritte:**
1. Einloggen
2. Erstmalig eine Hauptseite (z.B. Kulturen) aufrufen
3. Nach ~2 Sekunden erscheint unten eine Info-Snackbar

**Erwartetes Verhalten:**  
Der Tipp-Text kommt aus dem Übersetzungssystem (t()).

**Tatsächliches Verhalten:**  
Der Banner-Text ist hardcoded Deutsch:
```
💡 Tipp: Drücke Alt+K für die Command Palette.
```

**Code-Fundstelle:** `frontend/src/commands/CommandProvider.tsx:293`

```tsx
<Alert severity="info" onClose={() => setHintOpen(false)} closeText={t('common:actions.close')}>
  💡 Tipp: Drücke Alt+K für die Command Palette.
</Alert>
```

Bemerkenswert: `closeText={t('common:actions.close')}` im selben Element verwendet korrekt `t()`, der Haupttext jedoch nicht.

### BUG-L02 – RegisterPage: Fehlende autocomplete-Attribute auf Passwort-Feldern (Regression)

**Priorität:** Low  
**Bereich:** Registrierung / Passwort-Manager-Kompatibilität  
**Status:** Regression – war in `qa-report-2026-06-20-new-user.md` als Problem dokumentiert

**Reproduktionsschritte:**
1. `/register` aufrufen
2. `autocomplete`-Attribut der Passwort-`<input>`-Elemente prüfen (DevTools → Elements)

**Erwartetes Verhalten:**  
- Passwort-Feld: `autocomplete="new-password"`
- Passwort-Bestätigen-Feld: `autocomplete="new-password"`

**Tatsächliches Verhalten:**  
Beide Felder haben `autocomplete=null` (kein Attribut gesetzt).

**Konsequenz:** Passwort-Manager können Registrierungsformulare schwerer korrekt befüllen und das Speichern des neuen Passworts ist weniger verlässlich.

**Vermutete Ursache:**  
`slotProps.input.inputProps` mit `autoComplete="new-password"` fehlt auf beiden `<TextField>`-Komponenten in `RegisterPage.tsx`.

## 6. UX-Probleme

### UX-01 – Projekteinstellungen auf Desktop schwer auffindbar

**Bereich:** Navigation / Desktop

Die Projekteinstellungen sind auf dem Desktop über den Projektnamen-Dropdown im Header erreichbar. Dieser Weg ist für neue Benutzer nicht intuitiv, weil:
- Das Element visuell wie ein Projektwechsler aussieht (nicht wie ein Settings-Einstieg)
- Das Desktop-Mehr-Menü (⋮) **keinen** Eintrag für Projekteinstellungen enthält
- Im Gegensatz dazu hat das Mobile-Mehr-Menü direkt "Projekteinstellungen" als Menüpunkt

**Beobachtete Menüeinträge (Desktop-Mehr-Menü):**  
`Versionsverlauf öffnen | Kontoeinstellungen | Tastenkürzel | App-Hilfe | Abmelden`

Neue Desktop-Nutzer, die nach Projekteinstellungen suchen, werden das Mehr-Menü zuerst öffnen — und dort nichts finden.

---

### UX-02 – Kein Hinweis für Nicht-Admins auf Projekteinstellungen-Seite

**Bereich:** Projekteinstellungen / Mitgliederverwaltung

Die Mitgliederverwaltungs-Sektion auf `/app/project-settings` wird nur für Admins angezeigt (`canManageMembers = isProjectAdmin`). Für normale Mitglieder fehlt die Sektion vollständig — ohne Erklärung.

**Problem:** Ein Nicht-Admin-Mitglied, das nach der Mitgliederverwaltung sucht, sieht nur die Projektumbenennungs-Sektion und einen leeren Rest der Seite. Es gibt keinen Hinweis wie „Nur Admins können Mitglieder verwalten". Der Nutzer weiß nicht, ob die Funktion fehlt oder ob er keine Berechtigung hat.

## 7. Inkonsistenzen

### INC-01 – Desktop- und Mobile-Mehr-Menü enthalten unterschiedliche Einträge

**Bereich:** Navigation

Das Desktop-Mehr-Menü (⋮) und das Mobile-Mehr-Menü (⋮) haben strukturell unterschiedliche Inhalte:

| Eintrag | Desktop | Mobile |
|---------|---------|--------|
| Projekt wechseln | ❌ (nur im Header) | ✅ |
| Projekteinstellungen | ❌ (nur im Projektwechsler-Dropdown) | ✅ |
| Neues Projekt | ❌ | ✅ |
| Versionsverlauf öffnen | ✅ | ✅ |
| Kontoeinstellungen | ✅ | ✅ |
| Tastenkürzel | ✅ | ✅ |
| App-Hilfe | ✅ | ✅ |
| Abmelden | ✅ | ✅ |

Mobile-Nutzer finden alle Projektaktionen zentral im Mehr-Menü. Desktop-Nutzer müssen wissen, dass der Header-Bereich (Projektwechsler) für Projektaktionen zuständig ist. Die Diskrepanz kann bei Wechsel zwischen Viewports verwirren.

## 9. Verbesserungsvorschläge

### VS-01 – 404-Seite für unbekannte Routen

Eine React-Router-Catch-All-Route (`path="*"`) unterhalb von `/app/` mit einer benutzerfreundlichen "Seite nicht gefunden"-Meldung und einem Link zur Startseite würde die Robustheit erheblich verbessern. (Entspricht BUG-M01.)

### VS-02 – RegisterPage: Korrekte i18n und autocomplete

Beide Passwort-Toggle-Buttons sollten `t('auth:register.showPassword')`/`t('auth:register.hidePassword')` verwenden (analog zu LoginPage), und beide Passwort-Felder sollten `autocomplete="new-password"` tragen. (Entspricht BUG-M02 und BUG-L02.)

### VS-03 – Projekteinstellungen im Desktop-Mehr-Menü zugänglich machen

Ein Eintrag "Projekteinstellungen" im Desktop-Mehr-Menü (zusätzlich zum bestehenden Projektwechsler-Dropdown) würde die Auffindbarkeit für Desktop-Nutzer deutlich verbessern und die Konsistenz mit dem Mobile-Menü herstellen.

### VS-04 – Hinweis für Nicht-Admins auf Projekteinstellungen-Seite

Wenn `canManageMembers === false`, sollte die Mitglieder-Sektion statt leer zu sein eine kurze Meldung anzeigen: z.B. „Nur Projekt-Admins können Mitglieder verwalten." Dies verhindert Verwirrung bei nicht-administrativen Mitgliedern.

### VS-05 – CommandProvider-Tipp über t() übersetzen

Der Hinweistext `'💡 Tipp: Drücke Alt+K für die Command Palette.'` sollte in das i18n-System überführt werden, um Konsistenz mit dem Rest der Codebasis zu gewährleisten und eine eventuelle Mehrsprachigkeit zu ermöglichen. (Entspricht BUG-L01.)
